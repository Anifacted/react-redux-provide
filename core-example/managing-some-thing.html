Managing Some "Thing"

Managing Some "Thing"

Suppose we have a component that renders some thing. The component will extract thing from its props (properties) and render it. We'll also add thing to the component's propTypes so that we know what type of thing the component expects.

// src/components/SomeThing.js

import React, { PropTypes } from 'react';

const SomeThing = ({ thing }) => (
  <div>
    Thing is currently: {thing}
  </div>
);

SomeThing.propTypes = {
  thing: PropTypes.string.isRequired
};

export default SomeThing;

And then suppose we want another component to be able to set the current state of thing. We'll use a function called setThing. This is also extracted from the component's props, and we'll add an expected setThing function to its propTypes. We'll render a button that prompts the user for the next thing when clicked.

// src/components/SomeThingSetter.js

import React, { PropTypes } from 'react';

const SomeThingSetter = ({ setThing }) => (
  <button onClick={() => {
    const thing = window.prompt(`What should thing be?`);
    setThing(thing);
  }}>
    Click me to set thing.
  </button>
);

SomeThingSetter.propTypes = {
  setThing: PropTypes.func.isRequired
};

export default SomeThingSetter;

Now we need to create a provider that can manage the state of thing. For this example, all we need is a setThing action, and a thing reducer.

Let's break it down, step by step.

We'll create a value to represent the type of action taking place. You can create any number of values to represent any number of action types. These values will remain constant, which is why they are typically called constants. For this example, we need just one constant. Let's call it SET.

const SET = 'SET';

Now we need to create the setThing function used within the SomeThingSetter component above. This function will accept a new thing as an input, and it will return an object containing the new thing along with the SET action type.

function setThing(thing) {
  return {
    type: SET,
    thing
  };
}

We can condense this function into a single line:

const setThing = (thing) => ({ type: SET, thing });

The object returned by the setThing function describes an action taking place. A provider typically contains an actions object. The actions object can contain any number of functions like this.

With that said, let's create an object called actions and put our setThing function in it.

const actions = {
  setThing: (thing) => ({ type: SET, thing })
};

Now we need to create a function that might return a new state for some thing. This function should accept the current state and the action object as inputs. It should always return either the current state of thing or a new state. It will return a default thing if the current state is undefined. And it will return the new thing if the action type matches our SET constant.

function thing(state, action) {
  if (action.type === SET) {
    return action.thing;
  } else if (typeof state === 'undefined') {
    return 'nothing';
  } else {
    return state;
  }
}

We can also condense this function into a single line:

const thing = (state = 'nothing', { type, thing }) => type === SET ? thing : state;

This function is a reducer. A provider typically contains a reducers object to go with the actions object. The reducers object can contain any number of functions like this. Each reducer should be a pure function designed to return some state based on any action type.

With that said, let's create an object called reducers and put our thing function in it.

const reducers = {
  thing: (state = 'nothing', { type, thing }) => type === SET ? thing : state
};

These sets of actions and reducers become our provider module. All that's left now is to export these actions and reducers within a single object.

// src/providers/thing.js

const SET = 'SET';

const actions = {
  setThing: (thing) => ({ type: SET, thing })
};

const reducers = {
  thing: (state = 'nothing', { type, thing }) => type === SET ? thing : state
};

export default { actions, reducers };

You may have noticed the following:

  • The SomeThing component's propTypes contains thing, which matches the name of our reducer function.

  • The SomeThingSetter component's propTypes contains setThing, which matches the name of our action function.

Matching propTypes is how react-redux-provide knows that it should create a shared provider instance for these components. The SomeThingSetter component will automatically receive the setThing function. And the SomeThing component will automatically receive the current state of thing.

Note that components don't receive the same exact actions and reducers functions. They receive action creator functions and current states for matching keys. The action creator functions will automatically dispatch (pass) the action object to each reducer. All you should typically care about is the action objects and the states returned by each reducer.

Also note that an internal cache of reducer keys to components allows updates to be as efficient as possible. The fastest code is code that never runs!
