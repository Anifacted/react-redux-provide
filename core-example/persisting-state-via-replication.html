Persisting State via Replication

Persisting State via Replication

Suppose you want to save and/or share the current state of each thing whenever it changes. Replication allows you to do this in a way that keeps your application logic independent from your data sources. It allows you to:

  • Instantly swap one data source for another.
  • Persist state locally.
  • Persist state to some backend via some API.
  • Share state in real-time via websockets or other means.
  • Do each of the above at the same time.

In More Than One "Thing", we discussed the purpose of the provider's key function. The value returned by this function is accessible via store.key. Replicators can use the store and its key to save/share actions and state changes. Replicators are like providers in that they're a composition of functions. Replicators are typically designed to work with any provider.

Suppose you want to get the initial state of each thing from an API. A replicator's getInitialState function will receive the store along with a setState function and a setError function. Let's use this to create our own getInitialState function which will fetch the current state from our API.

const getInitialState = ({ store, setState, setError }) => (
  fetch(`/api/${store.key}`)
    .then(response => response.json())
    .then(setState)
    .catch(setError)
);

For example, if thingId is "A", then the store.key would be things/A and this getInitialState function would fetch the state of thing "A" from the API via the URL /api/things/A. It then attempts to convert the response to JSON which becomes the initial state via setState. If there's an error, it's caught by setError instead.

Replicating state changes to the API works similarly. The onStateChange function receives the same properties as getInitialState plus some others (see full API reference). For our particular case, we need nextState and create. The create property will be a function when creating a new provider instance. This is useful when the client relies on the server's state. For example, the client can't create and cache the correct store.key without the server returning a new thingId. The easiest way to achieve this is by calling the thunk API's createInstance function.

const onStateChange = ({ store, nextState, create, setState, setError }) => (
  fetch(`/api/${store.key}`, {
    method: create ? 'POST' : 'PATCH',
    body: JSON.stringify(nextState),
    headers: {
      'Accept': 'application/json',
      'Content-Type': 'application/json'
    }
  })
    .then(response => response.json())
    .then(create || setState)
    .catch(setError)
);

If we're creating a new thing instance, the store.key will be things/, which means we'll POST the nextState to /api/things/. We expect the server to return JSON containing any missing state - e.g., thingId - which the create function handles.

If we're not creating a new thing instance, we're just replicating the new state to the API. Suppose the state of thing "A" has changed. This onStateChange function would send the nextState to the server as a PATCH request to the URL /api/things/A.

Putting these two functions together will form our replicator:

// src/replicators/backend.js

const getInitialState = ({ store, setState, setError }) => (
  fetch(`/api/${store.key}`)
    .then(response => response.json())
    .then(setState)
    .catch(setError)
);

const onStateChange = ({ store, nextState, create, setState, setError }) => (
  fetch(`/api/${store.key}`, {
    method: create ? 'POST' : 'PATCH',
    body: JSON.stringify(nextState),
    headers: {
      'Accept': 'application/json',
      'Content-Type': 'application/json'
    }
  })
    .then(response => response.json())
    .then(create || setState)
    .catch(setError)
);

export default { getInitialState, onStateChange };
